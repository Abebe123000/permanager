# permanager ユーザーストーリー

## 概要

`permanager` は、ソースコード内に埋め込まれた仕様書へのパーマネントリンクを管理するための CLI ツールです。

## 背景

### コードと仕様の対応を明示することの重要性

ソフトウェア開発において、実装コードはほぼ必ず何らかの仕様・要件に基づいて書かれる。しかし多くの場合、「このコードがどの仕様に基づいているか」はコード中に明示されておらず、実装者の記憶やドキュメントの散在した記述に依存している。

これが問題になる場面は多い。

**新規参加者がコードを読む場合**: なぜこの実装になっているのかを理解するために、仕様書・設計書・チケット・チャットの履歴など複数のソースを横断して調べる必要がある。どこに根拠があるのかが自明でないため、理解に時間がかかるか、誤った前提で読み進めることになる。

**AI がコードを読む場合**: コードレビューや実装支援を行う AI にとっても同様の問題がある。仕様と実装の一致確認を行うには、どの仕様書を参照すべきかを明示的に与えられなければ、正確な判断ができない。コードの意図が仕様に沿っているかどうかを機械的に検証することも困難になる。

**仕様が変わった場合**: 仕様変更の影響範囲を把握するためには、変更された仕様を参照している実装を特定する必要がある。しかしそのつながりがコード中に記録されていなければ、影響を受ける箇所を見つけることは人間にも AI にも難しい。

### パーマネントリンクによる解決

これらの問題に対するシンプルなアプローチが、コードに仕様書へのパーマネントリンクを直接埋め込むことである。Git のパーマネントリンク（特定コミットを指す URL）を使えば、実装時点の仕様内容を正確に参照できる。

```rust
// この関数は以下の仕様に基づいて実装されています
// See: https://github.com/owner/repo/blob/abc123/docs/api-spec.md#L15-L30
fn validate_request(req: &Request) -> Result<(), Error> {
    // ...
}
```

コードとその根拠となる仕様が明示的に紐づくことで、人間も AI も「この実装は何に基づいているか」を即座に把握できるようになる。

## パーマネントリンク運用によって可能になること

パーマネントリンクが存在することで、コードと仕様書の間に2方向のナビゲーションが生まれる。

### コード → 仕様書

コードを読む人間や AI が、その実装の根拠となる仕様を即座に参照できる。リンクをたどるだけでよく、git や GitHub などの既存ツールで十分に機能する。専用ツールがなくてもこの方向のメリットは享受できる。

### 仕様書 → コード

ある仕様書の変更が、どのコードに影響するかを追跡できる。これにより以下が可能になる：

- 仕様書の変更差分と、その仕様を参照しているコードの対応を把握できる
- 「仕様が変わったのに実装が追従していない」状態を検出できる
- 仕様変更の影響範囲を網羅的に把握できる

ただしこの方向は、リンクが存在するだけでは実現できない。全コードベースから特定の仕様を参照している箇所を収集し、参照先のコミットと現在の仕様を比較するためには、専用ツールが必要になる。

## パーマネントリンク運用の課題

### 1. コード修正時にリンクが適切に更新されているか

実装を変更した際、参照すべき仕様が変わっていればリンクも更新する必要がある。逆に、リンクが古いままになっていても静的には検出できない。

この確認作業は、AI によるコードレビューで十分に対応できる。コード差分とリンク先の仕様を照合し、「このコード変更に対してリンクの更新が必要かどうか」を判断することは、現在の AI の能力の範囲内にある。

### 2. 仕様書より先にコードが更新される場合

仕様書がまだ存在しない、あるいは仕様がコードより後から確定するケースでは、一時的にリンクが存在しない状態や、暫定的な箇所を指した状態になる。

これ自体は避けられない運用上の現実であり、問題は「仕様書が後から更新されたときに、対応するコードへの追従漏れが起きること」にある。仕様書更新のタイミングで影響コードを特定し、リンクを正しい箇所に更新できれば、この課題は解消できる。

## 専用ツールが提供すべき機能

機械的に一意に定まる情報の提供は専用ツールが担い、その情報をもとに意味的な判断を行うのは AI が担う、という役割分担が望ましい。

### ツールが提供すべきもの

以下はいずれも、コードと仕様書の内容を機械的に解析すれば一意に答えが出る情報である。

- **参照関係の一覧**: どのコードのどの行が、どの仕様書のどのコミット・行を参照しているか
- **仕様変更の影響範囲**: 仕様書のある変更に対して、それを参照しているコードの箇所はどこか
- **リンクの陳腐化の検出**: あるリンクが指すコミットから現在の仕様書 HEAD までの間に差分があるかどうか

### AI が担うべきもの

ツールが提供する情報を受け取ったうえで、以下のような判断は AI が行う。

- 仕様の変更内容がコードに与える意味的な影響の評価
- 実装が仕様に沿っているかどうかの確認
- リンクの更新や実装修正が必要かどうかの判断と提案
